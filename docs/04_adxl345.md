# Milestone 4 — ADXL345 Raw Acceleration

## Goal

Verify the connected GY-291/ADXL345, read raw X/Y/Z acceleration, and use the
X/Y values as tilt input for the particle animation.

This milestone does not add shake detection, motion states, calibration
storage, another RTOS task, or TinyML.

## Verified wiring

| GY-291 pin | ESP32-C3 connection | Purpose |
|---|---|---|
| `VCC` | `3V3` | safe module power |
| `GND` | `GND` | common electrical reference |
| `SDA` | `GPIO5` | I2C data |
| `SCL` | `GPIO6` | I2C clock |
| `CS` | `3V3` | selects I2C mode |
| `SDO` | `GND` | selects address `0x53` |
| `INT1`, `INT2` | not connected | interrupts are deferred |

The I2C scanner found one device at address `0x53` on 2026-06-24.

## Device verification

The driver reads register `DEVID` (`0x00`) before configuring the sensor. A
real ADXL345 must return `0xE5`. An ACK at address `0x53` proves that an I2C
device exists; `DEVID = 0xE5` proves that the responding device identifies as
an ADXL345.

## Minimal configuration

The first implementation uses:

* full-resolution mode
* ±2 g range
* 100 Hz output data rate
* measurement mode enabled through `POWER_CTL`

Acceleration data is read from six consecutive registers beginning at
`DATAX0`. Each axis is a signed 16-bit little-endian value.

In full-resolution mode, approximately 256 counts represent 1 g:

```text
acceleration_g = raw_value / 256
```

The exact zero offset and orientation will be calibrated later.

## Component boundary

```text
main
├── i2c_bus: owns the ESP-IDF I2C master bus
├── adxl345: verifies and reads the sensor
├── fluid_animation: consumes normalized tilt
└── oled_display: sends the framebuffer over SPI
```

`fluid_animation` does not include `adxl345.h`. This keeps animation physics
independent from the selected sensor model.

## Expected monitor output

```text
I (...) i2c_bus: Found I2C device at address 0x53
I (...) adxl345: ADXL345 initialized: address=0x53, DEVID=0xE5
I (...) keychain: ADXL345 raw: X=..., Y=..., Z=...
I (...) keychain: Starting ADXL345-controlled particle animation
```

When the module is approximately flat, X and Y should be near zero and Z
should have a magnitude near 256 counts. Tilting the module changes the axis
values. Axis signs depend on module orientation and will be verified on the
real prototype.

The cellular animation uses tilt magnitude to choose how often a particle may
advance by one pixel. It does not convert acceleration into continuous pixel
velocity.

## Acceptance criteria

* the project builds for ESP32-C3
* scanner still reports `0x53`
* `DEVID` is `0xE5`
* raw X/Y/Z values appear periodically in the serial monitor
* values change when the module is tilted
* particle motion responds to X/Y tilt
* particles remain inside all four display boundaries
* the SPI OLED continues operating while I2C remains active

## Deferred work

* sensor offset calibration
* configurable axis inversion and board orientation
* shake and stillness detection
* FLUID/SLEEP/TIME state machine
* ADXL345 interrupt pins
* optional liquid-specific viscosity and cohesion after the granular test
* RTOS application tasks and TinyML

## Manual exercise

Place the module flat and record X/Y/Z. Then tilt it separately toward the X
and Y arrows and note which value changes and whether its sign is positive or
negative.

## Review questions

1. Why is reading `DEVID` stronger evidence than receiving only an I2C ACK?
2. Why are two bytes required for each acceleration axis?
3. Why does a flat sensor normally report gravity mainly on the Z axis?
4. Why does the animation receive normalized tilt instead of raw sensor bytes?
5. Why does the ADXL345 component borrow rather than own the I2C bus?
