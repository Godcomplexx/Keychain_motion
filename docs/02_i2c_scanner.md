# Milestone 2 — I2C Scanner

## 1. Goal

Create a small I2C scanner for the ESP32-C3 Smart Motion Keychain.

The scanner will verify that the ESP32-C3 can communicate with the two devices
planned for the shared I2C bus:

* 0.96 inch 128x64 I2C OLED display
* ADXL345 GY-291 accelerometer module

This milestone is complete when the serial monitor reports the addresses of the
connected devices. It does not draw on the OLED or read motion data.

---

## 2. What I Need to Understand

Before implementing the scanner, I need to understand:

* that I2C is a shared communication bus
* the roles of SDA and SCL
* how the controller selects a device by address
* how a device acknowledges a request
* why pull-up resistors are required
* why the GPIO mapping belongs in `board_config.h`
* why communication must be verified before device drivers are written

### ESP-IDF terminology

In this project, the ESP32-C3 is the **I2C master**. It starts communication and
chooses which device to contact. The original OLED and ADXL345 are **I2C devices** that
respond when the master sends their addresses.

The future `i2c_bus` component will own bus initialization and scanning. The
application entry point will only call that component and report the result, so
`main.c` stays small.

---

## 3. Beginner Explanation of I2C

I2C, pronounced "I-squared-C," is a way for a microcontroller to communicate
with several chips using only two signal wires.

The ESP32-C3 starts a transaction, sends the address of the device it wants to
contact, and waits for an acknowledgement. A matching device responds with an
**ACK**. If no device responds, the controller receives a **NACK** or timeout.

An I2C scanner repeats this address check across the usable 7-bit address range.
It does not need to know whether a responding device is an OLED or a sensor. It
only reports which addresses acknowledged.

I2C signal lines use pull-up resistors. Many OLED and GY-521 breakout boards
already include them, but this must be checked on the actual modules. The bus
must also have a common ground, and its signal voltage must be safe for the
ESP32-C3.

---

## 4. What SDA and SCL Are

### SDA — Serial Data

SDA carries addresses, commands, and data between the ESP32-C3 and I2C devices.
Communication on SDA can travel in both directions.

### SCL — Serial Clock

SCL carries the clock signal generated by the I2C master. The clock tells every
device when each bit on SDA should be read.

All devices on one bus connect to the same SDA and SCL lines:

```text
ESP32-C3 SDA ----+---- OLED SDA
                 +---- ADXL345 SDA

ESP32-C3 SCL ----+---- OLED SCL
                 +---- ADXL345 SCL

ESP32-C3 GND ----+---- OLED GND
                 +---- ADXL345 GND
```

The final SDA and SCL GPIO numbers must be selected from the exact ESP32-C3
board pinout and stored in `board_config.h`. They must not be copied from an
unrelated board or hardcoded inside the scanner.

---

## 5. What an I2C Address Is

An I2C address is a small number that identifies a device on the shared bus.
This project will use 7-bit addresses, normally written in hexadecimal form
such as `0x3C` or `0x68`.

Common addresses for the planned modules are:

* OLED display: usually `0x3C`, sometimes `0x3D`
* ADXL345: `0x53` when SDO is low, or `0x1D` when SDO is high

These are expected values, not proof of the actual wiring. Module variants,
solder jumpers, and the ADXL345 SDO pin can change an address. The scanner must
measure what is really present.

The scanner should check the normal usable 7-bit range from `0x08` through
`0x77`. Addresses outside that range are reserved for special I2C purposes and
should not be probed in this beginner milestone.

---

## 6. Why Multiple Devices Can Share One Bus

The original OLED and ADXL345 could use the same SDA and SCL wires because the ESP32-C3
addresses one device at a time.

For example:

1. The ESP32-C3 sends address `0x3C` to communicate with the OLED.
2. The ADXL345 sees that the address does not match and stays inactive.
3. The ESP32-C3 later sends address `0x53` to communicate with the ADXL345.
4. The OLED sees that the address does not match and stays inactive.

Sharing works as long as:

* each active device has a unique address
* every device uses a compatible signal voltage
* SDA, SCL, power, and ground are wired correctly
* the bus has suitable pull-up resistors
* the selected bus speed is supported by every device

---

## 7. Why the Scanner Comes Before Device Drivers

An OLED driver assumes that the display is powered, wired correctly, and
responding at a known address. An ADXL345 driver makes the same assumptions
about the sensor.

If a complete driver fails immediately, several causes are possible:

* wrong SDA or SCL GPIO
* swapped SDA and SCL wires
* missing common ground
* wrong address
* device not powered
* unsafe or incorrect voltage
* missing pull-up resistors
* faulty cable, breadboard connection, or module
* software defect in the driver

A scanner removes much of this uncertainty. It tests only the shared bus and
address acknowledgement. Once the expected addresses appear, later OLED and
ADXL345 problems can be investigated inside their own drivers.

This milestone therefore verifies the foundation before more code is added.

---

## 8. Expected Serial Monitor Output

With both devices connected correctly, output should look similar to:

```text
I (...) i2c_bus: Starting I2C scan
I (...) i2c_bus: Found I2C device at address 0x3C
I (...) i2c_bus: Found I2C device at address 0x68
I (...) i2c_bus: I2C scan complete: 2 device(s) found
```

The exact device addresses may differ. If nothing acknowledges, the
output should clearly report that result:

```text
I (...) i2c_bus: Starting I2C scan
W (...) i2c_bus: I2C scan complete: no devices found
```

The scanner should continue past addresses that do not respond. A NACK during
a scan is expected and is not automatically a fatal error.

## 10. Mistakes to Avoid

### Inventing GPIO numbers

Do not choose SDA and SCL randomly. Some GPIOs may be used for bootstrapping,
USB, flash, or board-specific functions. Verify the exact board pinout first.

### Hardcoding pins in the scanner

Do not place raw GPIO numbers inside `i2c_bus.c` or `main.c`. Store the hardware
mapping in `board_config.h` so the later switch from a temporary ESP32-C6 board
to ESP32-C3 remains controlled.

### Confusing 7-bit and 8-bit addresses

Scanner output should use 7-bit addresses such as `0x3C` and `0x68`. Some
datasheets show an address combined with the read/write bit, producing a
different-looking 8-bit value. Do not shift an already 7-bit address twice.

### Treating every NACK as a fatal error

Most scanned addresses will not contain a device and will return NACK. That is
normal. Initialization failures and unexpected bus errors should be handled
separately.

### Using unsafe voltage

Do not allow SDA or SCL pull-ups to raise the signal lines above a voltage safe
for the ESP32-C3. Do not assume every breakout board handles voltage the same
way.

### Forgetting common ground

SDA and SCL signals need a shared electrical reference. Connect the grounds of
the ESP32-C3, OLED, and ADXL345.

### Adding device logic too early

Do not draw pixels, initialize an OLED controller, read ADXL345 registers, add
RTOS tasks, or add TinyML during this milestone.
